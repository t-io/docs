## Overview

The RolescopeAI Developer API lets you integrate CV analysis directly into your
ATS, recruitment platform, or internal tools. Upload candidate CVs, define
position criteria, and receive structured scoring results — the same forensic
analysis pipeline that powers the dashboard.

**Base URL**

```
https://api.rolescope.ai/api/v1/developer
```

## Authentication

All API requests require an API key passed in the `Authorization` header.

```bash
Authorization: Bearer rsk_live_a1b2c3d4e5f6...
```

There are two auth modes depending on the endpoint:

| Mode | Header | Used for |
|------|--------|----------|
| **API Key** | `Bearer rsk_...` | All data endpoints (positions, candidates, credits, webhooks) |
| **JWT** | `Bearer eyJ...` | Key management (`POST /keys`, `GET /keys`, `DELETE /keys`) and credit checkout |

<Info>
Most integrations only use API key auth. JWT auth is only needed for
managing the keys themselves (creating, listing, revoking) and purchasing credits —
both of which can also be done from the dashboard.
</Info>

### Getting your API key

<Steps>
  <Step title="Open Settings">
    Log in to [app.rolescope.ai](https://app.rolescope.ai) and navigate to
    **Settings → Developer**.
  </Step>
  <Step title="Create a key">
    Click **Create API Key**, give it a name (e.g. "ATS Integration"), and
    optionally set an expiration date.
  </Step>
  <Step title="Copy immediately">
    The full key (starting with `rsk_`) is shown **once**. Copy it and store it
    securely. You will not be able to see it again — only the prefix is stored.
  </Step>
  <Step title="Use it in requests">
    Pass the key in the `Authorization: Bearer rsk_...` header on every API call.
  </Step>
</Steps>

<Warning>
Treat your API key like a password. Do not commit it to version control or
expose it in client-side code. Use environment variables or a secrets manager.
</Warning>

## Quickstart

A minimal end-to-end flow: create a position, upload a CV, poll for results.

<Steps>
  <Step title="Create a position">
    ```bash
    curl -X POST https://api.rolescope.ai/api/v1/developer/positions \
      -H "Authorization: Bearer $RSK_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{
        "name": "Senior Backend Engineer",
        "skills": ["Python", "FastAPI", "PostgreSQL"],
        "min_experience_months": 36,
        "location": "Berlin",
        "work_mode": "remote",
        "languages": ["English", "German"]
      }'
    ```
  </Step>
  <Step title="Get an upload URL">
    ```bash
    curl -X POST https://api.rolescope.ai/api/v1/developer/files/upload-url \
      -H "Authorization: Bearer $RSK_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"filename": "resume.pdf"}'
    ```
  </Step>
  <Step title="Upload the PDF">
    Upload the file directly to the `upload_url` returned in the previous step.
    ```bash
    curl -X PUT "$UPLOAD_URL" \
      -H "Content-Type: application/pdf" \
      --data-binary @resume.pdf
    ```
  </Step>
  <Step title="Submit the candidate">
    ```bash
    curl -X POST https://api.rolescope.ai/api/v1/developer/positions/$POSITION_ID/candidates \
      -H "Authorization: Bearer $RSK_API_KEY" \
      -H "Content-Type: application/json" \
      -d '{"file_id": "'$FILE_ID'", "label": "Jane Doe"}'
    ```
  </Step>
  <Step title="Poll for results">
    ```bash
    curl https://api.rolescope.ai/api/v1/developer/jobs/$JOB_ID \
      -H "Authorization: Bearer $RSK_API_KEY"
    ```
    Repeat until `status` is `"completed"` or `"failed"`.
  </Step>
  <Step title="Get the analysis">
    ```bash
    curl https://api.rolescope.ai/api/v1/developer/candidates/$CANDIDATE_ID \
      -H "Authorization: Bearer $RSK_API_KEY"
    ```
  </Step>
</Steps>

### API Keys

<Note>
Key management endpoints require **JWT auth** (admin role), not API key auth.
These operations can also be performed from the dashboard under Settings → Developer.
</Note>

#### Create API key

```
POST /keys
```

<ParamField body="name" type="string" required>
  A descriptive name for the key (e.g. "Production ATS").
</ParamField>
<ParamField body="expires_at" type="string">
  Optional ISO 8601 expiration timestamp. Omit for non-expiring keys.
</ParamField>

</ResponseField>

#### List API keys

```
GET /keys
```

Returns all active (non-revoked) API keys. The full key is **never** returned — only the prefix.

#### Revoke API key

```
DELETE /keys/{key_id}
```

Permanently revokes a key. Returns `204 No Content` on success.

### Webhook Payload

When an event fires, RolescopeAI sends a `POST` request to your webhook URL.

**Headers**

| Header | Description |
|--------|-------------|
| `Content-Type` | `application/json` |
| `X-RolescopeAI-Signature` | `sha256=<hex-digest>` — HMAC-SHA256 of the request body using your webhook secret |
| `X-RolescopeAI-Event` | Event type: `job.completed` or `job.failed` |

**Payload**

```json
{
  "event": "job.completed",
  "timestamp": "2026-04-02T14:30:00Z",
  "data": {
    "job_id": "abc-123",
    "type": "analyze_candidate",
    "status": "completed",
    "position_id": "pos-456",
    "candidate_id": "cand-789",
    "label": "Jane Doe",
    "tokens_charged": 35000,
    "completed_at": "2026-04-02T14:30:00Z",
    "result": { }
  }
}
```

For `job.failed` events, the `data` object includes an `error` field instead of `result`.

#### Verifying webhook signatures

<CodeGroup>
```python Python
import hmac
import hashlib

def verify_signature(payload_bytes: bytes, secret: str, signature_header: str) -> bool:
    """Verify the X-RolescopeAI-Signature header."""
    expected = hmac.new(
        secret.encode(),
        payload_bytes,
        hashlib.sha256,
    ).hexdigest()
    received = signature_header.removeprefix("sha256=")
    return hmac.compare_digest(expected, received)
```

```javascript Node.js
const crypto = require("crypto");

function verifySignature(payloadBuffer, secret, signatureHeader) {
  const expected = crypto
    .createHmac("sha256", secret)
    .update(payloadBuffer)
    .digest("hex");
  const received = signatureHeader.replace("sha256=", "");
  return crypto.timingSafeEqual(
    Buffer.from(expected),
    Buffer.from(received)
  );
}
```
</CodeGroup>

<Warning>
Always verify the signature before processing webhook payloads. Reject
requests with missing or invalid signatures.
</Warning>

## Error Handling

All error responses follow a consistent shape:

```json
{
  "detail": "Position not found"
}
```

Or for structured errors:

```json
{
  "detail": {
    "message": "Insufficient token balance",
    "code": "INSUFFICIENT_TOKENS"
  }
}
```

| Status | Meaning |
|--------|---------|
| `400` | Bad request — invalid parameters or request body |
| `401` | Unauthorized — missing or invalid API key |
| `402` | Payment required — insufficient token balance |
| `404` | Not found — resource doesn't exist or doesn't belong to your account |
| `422` | Validation error — request body failed schema validation |
| `500` | Internal server error |

## Token Costs

Each CV analysis consumes tokens from your balance. Tokens are **reserved**
when you submit a candidate and **finalized** when processing completes.
If analysis fails, reserved tokens are refunded.

| Credit Pack | Tokens | Price |
|-------------|--------|-------|
| Starter | 300,000 | €30 |
| Growth | 550,000 | €50 |
| Scale | 1,200,000 | €100 |

<Tip>
Use the `GET /credits` endpoint to monitor your balance, or set up a webhook
to track `job.completed` events which include the `tokens_charged` field.
</Tip>

## Rate Limits

The API is rate-limited per account. Current limits:

| Scope | Limit |
|-------|-------|
| General requests | 100 req/min |
| Candidate uploads | 50 req/min |
| Batch uploads | 10 req/min |

Rate-limited responses return `429 Too Many Requests` with a `Retry-After` header.
